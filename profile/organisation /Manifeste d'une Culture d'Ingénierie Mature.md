# Manifeste d'une Culture d'Ingénierie Mature

## Préambule

L'ingénierie logicielle n'est ni une démocratie absolue ni une hiérarchie de vérité.

Une organisation technique mature reconnaît la valeur de l'expérience, tout en refusant l'argument d'autorité. Les décisions doivent être guidées par les faits, la qualité du raisonnement, l'expérimentation et la responsabilité, plutôt que par le rang, l'ancienneté ou le statut.

L'objectif collectif n'est pas de gagner les débats, mais de construire les meilleures solutions possibles tout en maîtrisant les risques.

---

# 1. Principes fondamentaux

## 1.1 La vérité n'a pas de grade

Aucun titre, niveau d'ancienneté ou position hiérarchique ne garantit qu'une personne a raison.

Les idées doivent être évaluées selon :

* leur cohérence ;
* les données disponibles ;
* les contraintes du système ;
* leur capacité à être testées et réfutées ;
* leurs conséquences opérationnelles.

Une proposition correcte reste correcte, quel que soit son auteur.

---

## 1.2 L'expérience possède une valeur, mais pas une infaillibilité

L'expérience apporte :

* la reconnaissance des schémas d'échec ;
* la connaissance du contexte historique ;
* la compréhension des compromis ;
* l'identification des risques cachés.

Cependant :

L'expérience ne constitue jamais une preuve.

Les arguments d'autorité tels que :

> « Je fais ce métier depuis vingt ans, donc j'ai raison »

sont incompatibles avec une culture d'ingénierie saine.

---

## 1.3 Responsabilité et vérité sont deux concepts distincts

L'autorité décisionnelle ne doit jamais être confondue avec l'autorité intellectuelle.

Le décideur :

* tranche lorsque cela est nécessaire ;
* assume les conséquences ;
* porte la responsabilité des risques.

Il n'est pas propriétaire de la vérité.

---

# 2. Rôle des différents niveaux d'expérience

## 2.1 Le rôle du Junior

Le Junior :

* apporte un regard neuf ;
* remet en question les hypothèses implicites ;
* contribue aux débats techniques ;
* apprend continuellement.

Son manque d'ancienneté ne réduit pas la valeur potentielle de ses idées.

---

## 2.2 Le rôle du Senior

Le Senior :

* fournit du contexte ;
* identifie les risques ;
* partage les erreurs déjà rencontrées ;
* facilite la prise de décision ;
* favorise la progression des autres ingénieurs.

Son rôle n'est pas de gagner les discussions mais d'améliorer la qualité des décisions.

---

## 2.3 Le rôle du Lead ou Principal Engineer

Le Lead :

* arbitre lorsque le consensus n'émerge pas ;
* protège la cohérence du système ;
* équilibre les contraintes techniques et métier ;
* assume la responsabilité des décisions.

La responsabilité supplémentaire ne confère aucune supériorité intellectuelle.

---

# 3. Principes de discussion technique

## 3.1 Critiquer les idées, jamais les personnes

Les débats doivent porter sur :

* les hypothèses ;
* les contraintes ;
* les données ;
* les compromis.

Les attaques personnelles, l'ego ou les rapports de force n'ont pas leur place dans une discussion technique.

---

## 3.2 L'expérience doit être expliquée

Une intuition issue de l'expérience est précieuse.

Elle doit cependant être exprimée sous forme de :

* retours d'expérience ;
* exemples d'incidents passés ;
* risques identifiés ;
* hypothèses vérifiables.

---

## 3.3 Changer d'avis est une force

Reconnaître une erreur n'est pas un signe de faiblesse.

La capacité à réviser son opinion face aux faits constitue l'une des qualités essentielles d'un ingénieur.

---

# 4. Culture de la preuve

Les décisions importantes doivent être fondées sur :

* des benchmarks ;
* des métriques ;
* des prototypes ;
* des expérimentations ;
* des retours de production ;
* des analyses de risques.

Les faits priment sur les opinions.

---

# 5. Documentation et décisions

## RFC (Request For Comments)

Les RFC permettent :

* d'exposer les différentes options ;
* de formaliser les hypothèses ;
* de recueillir les objections ;
* de documenter les compromis.

---

## ADR (Architecture Decision Record)

Les ADR servent à conserver :

* les décisions prises ;
* leur contexte ;
* les alternatives étudiées ;
* les raisons du choix effectué.

---

## POC (Proof of Concept)

Lorsqu'un débat ne peut être résolu théoriquement, l'expérimentation doit être privilégiée.

Le code et les résultats sont souvent plus instructifs que les opinions.

---

# 6. Gestion des désaccords

L'absence de consensus ne doit pas bloquer l'organisation.

Lorsqu'aucun accord n'émerge :

1. les arguments sont exposés ;
2. les risques sont identifiés ;
3. les données disponibles sont analysées ;
4. un responsable prend une décision ;
5. cette décision pourra être réévaluée ultérieurement.

Décider n'implique pas être infaillible.

---

# 7. Humilité intellectuelle

Personne :

* n'a toujours raison ;
* ne possède toute l'information ;
* n'est à l'abri de l'erreur.

L'humilité intellectuelle est une condition indispensable à l'amélioration continue.

---

# 8. Anti-principes

Une culture d'ingénierie mature rejette :

* les arguments d'autorité ;
* les guerres d'ego ;
* les décisions politiques ;
* les titres utilisés comme preuve ;
* la peur de remettre en question les choix établis ;
* la confusion entre ancienneté et compétence ;
* la recherche systématique du consensus au détriment de la vitesse de décision.

---

# 9. Objectif collectif

Le but d'une équipe d'ingénierie n'est pas :

* d'avoir raison ;
* de défendre son territoire ;
* de préserver son ego ;
* de gagner des débats.

Le but est :

* de comprendre les problèmes ;
* d'identifier les compromis ;
* de réduire les risques ;
* de construire des systèmes robustes ;
* d'améliorer continuellement les pratiques ;
* de créer de la valeur durable.

---

# Principe fondamental

> Les responsabilités sont hiérarchiques.
>
> La recherche de la vérité ne l'est pas.
>
> Les titres déterminent qui décide.
>
> Les faits déterminent ce qui est vrai.
>
> Une organisation d'ingénierie mature ne demande pas :
>
> « Qui a parlé ? »
>
> Elle demande :
>
> « Quels sont les arguments, quelles sont les preuves et quels sont les risques ? »
