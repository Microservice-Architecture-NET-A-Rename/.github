# Le pouvoir contre la compétence : critique du recrutement technique moderne

Le recrutement technique dans l’industrie logicielle souffre d’un paradoxe fondamental : les entreprises affirment rechercher de la compétence, mais leurs mécanismes d’évaluation mesurent souvent autre chose.

Ce décalage n’est pas marginal. Il est structurel.

Et il révèle une confusion profonde entre **évaluer une performance ponctuelle** et **évaluer une capacité réelle de production, d’évolution et d’adaptation**.

## L’erreur fondamentale : confondre état et trajectoire

La majorité des processus de recrutement technique reposent sur une logique simple : mesurer un niveau.

Un test.
Une série de questions.
Un exercice algorithmique.
Une épreuve chronométrée.

Ces mécanismes produisent une photographie.

Ils répondent à une question :

*"Que sait faire cette personne maintenant, dans ce cadre précis ?"*

Mais une entreprise ne recrute pas une photographie.

Elle recrute une trajectoire.

Elle recrute une personne qui va :

* apprendre ;
* s’adapter ;
* évoluer ;
* absorber de la complexité ;
* résoudre des problèmes nouveaux ;
* travailler avec d’autres.

Le problème fondamental est là :

**la valeur réelle d’un développeur n’est pas son état technique présent, mais sa dynamique d’évolution.**

Deux développeurs de niveau équivalent aujourd’hui peuvent avoir des trajectoires radicalement différentes demain.

L’un peut plafonner.
L’autre peut doubler sa valeur en deux ans.

Un bon recrutement cherche à prédire cette dynamique.

La majorité des processus classiques en sont incapables.

## Le faux signal des tests techniques

Le test technique moderne est devenu un standard industriel.

Mais un standard n’est pas nécessairement une bonne méthode.

La plupart de ces tests mesurent :

* rapidité d’exécution ;
* mémorisation de patterns ;
* résistance au stress ;
* familiarité avec un format.

Mais ils mesurent rarement :

* architecture logicielle ;
* maintenabilité ;
* capacité de refactoring ;
* compréhension systémique ;
* communication technique ;
* arbitrage entre contraintes.

Or le métier réel ne ressemble pas à un test.

Le métier réel consiste à :

comprendre un code legacy,
réduire de la dette technique,
arbitrer entre vitesse et qualité,
faire évoluer un système vivant.

Le test technique abstrait transforme une compétence contextuelle en performance artificielle.

Il remplace le réel par une simulation.

Et toute simulation introduit une perte de pertinence.

## Le problème de légitimité de l’évaluateur

Un autre problème fondamental apparaît dans beaucoup d’organisations :

qui évalue ?

Évaluer techniquement un développeur exige une proximité réelle avec la pratique.

Non pas historique.

Actuelle.

Un ancien développeur éloigné du terrain depuis des années, un manager non pratiquant, ou un chef de projet sans expertise technique active peut difficilement évaluer avec précision :

* la qualité des décisions techniques ;
* la pertinence architecturale ;
* la gestion de complexité.

Le problème ici n’est pas hiérarchique.

Il est épistémique.

**Qui possède une connaissance suffisamment valide pour produire un jugement fiable ?**

Quand l’évaluation est confiée à des profils éloignés de la pratique, l’organisation dégrade la qualité de son propre mécanisme de sélection.

Elle introduit du bruit décisionnel.

## L’illusion du modèle GAFAM

Beaucoup d’entreprises ont adopté les codes des grandes entreprises technologiques.

Tests algorithmiques.
Entretiens techniques.
Évaluations standardisées.

Mais elles ont copié la forme sans comprendre la structure.

Des entreprises comme Google, Meta ou Amazon n’utilisent pas ces méthodes pour vérifier une accumulation de connaissances.

Elles cherchent des signaux précis :

* vitesse d’apprentissage ;
* abstraction ;
* adaptation ;
* résolution de problèmes nouveaux.

Leur objectif est prédictif.

Pas scolaire.

Le problème de nombreuses entreprises plus classiques est qu’elles transforment ces outils en instruments de validation élitiste.

Le test devient un rite.

Pas un outil.

## Le rapport de force comme corruption de l’évaluation

L’évaluation peut devenir un espace de pouvoir.

Dans certains entretiens, le but implicite n’est plus de comprendre le candidat.

Le but devient :

tester sa résistance,
affirmer une position dominante,
produire une asymétrie symbolique.

Le candidat doit alors démontrer sa légitimité face à une autorité qui cherche moins à comprendre qu’à filtrer.

Cette logique est profondément contre-productive.

Pourquoi ?

Parce qu’un recrutement efficace repose sur l’alignement.

Pas sur la domination.

Une entreprise ne gagne rien à humilier un candidat compétent.

Elle gagne à comprendre :

comment il pense,
comment il apprend,
comment il collabore,
comment il construit.

Quand l’entretien devient un rapport de force, l’évaluation est corrompue.

## Le politique contre l’opérationnel

Dans certaines structures, le recrutement devient un acte politique.

Non politique au sens idéologique.

Politique au sens organisationnel.

Les acteurs impliqués cherchent à :

réduire leur risque,
préserver leur position,
maintenir leur influence.

Dans ce cadre, recruter un excellent profil peut devenir risqué.

Un profil très compétent peut :

remettre en question l’existant,
révéler les insuffisances,
déplacer les équilibres.

Le système préfère alors parfois le profil rassurant au profil excellent.

Le problème n’est donc plus la qualité du candidat.

Le problème est la structure de décision.

## Le dépôt GitHub comme signal ignoré

Un dépôt riche, structuré, vivant constitue souvent un signal beaucoup plus fort qu’un test standardisé.

Il permet d’observer :

* cohérence architecturale ;
* progression dans le temps ;
* diversité des problèmes traités ;
* style de code ;
* discipline de maintenance.

C’est du réel.

Mais ce réel exige du temps, de la compétence et de l’attention.

Le test standardisé est plus simple.

Moins coûteux.

Mais aussi souvent moins pertinent.

Le recrutement moderne privilégie fréquemment le signal facile au signal fort.

## Une organisation révèle sa qualité par sa manière d’évaluer

La manière dont une organisation recrute révèle sa maturité.

Une organisation mature cherche :

la prédiction juste.

Une organisation immature cherche :

la validation de ses propres représentations.

Si une entreprise confie l’évaluation technique à des profils inadaptés, utilise des signaux faibles comme outils principaux, et transforme l’entretien en rapport de force, alors le problème n’est pas le recrutement.

Le problème est organisationnel.

Le recrutement n’est qu’un symptôme.

## Conclusion

Le recrutement technique moderne souffre d’une dérive :

il mesure trop souvent la performance d’entretien au lieu de mesurer la capacité réelle de production et d’évolution.

Le problème n’est pas qu’il y ait des tests.

Le problème est ce que ces tests prétendent représenter.

Un développeur n’est pas une photographie.

C’est une trajectoire.

Et une entreprise qui ne sait pas reconnaître cette trajectoire ne recrute pas mal par accident.

Elle recrute mal par structure.
