### 1. Le désintérêt pour le “signal réel” (portfolio, dépôts, projets)

Un dépôt riche (GitHub, GitLab, projets persos, contributions open source) est souvent un signal bien plus fort qu’un CV.

Pourquoi ?

Parce qu’il montre :

* la continuité (vous codez réellement, pas seulement “dans le cadre du travail”) ;
* la capacité à terminer des projets ;
* les choix d’architecture ;
* la qualité du code ;
* la gestion de la dette technique ;
* l’évolution dans le temps.

Pourtant, beaucoup de recruteurs ou managers ne l’analysent pas, pour plusieurs raisons :

* manque de compétence technique pour l’évaluer ;
* manque de temps ;
* process RH standardisés ;
* préférence pour des signaux faciles à comparer (diplôme, années d’expérience, score à un test).

Le paradoxe : on ignore souvent le **travail réel** au profit de **proxies**.

---

### 2. L’évaluation technique par des profils non techniques (ou plus techniques)

Là aussi, c’est un problème fréquent.

Évaluer un développeur expérimenté demande de distinguer :

* maîtrise syntaxique ;
* maîtrise conceptuelle ;
* architecture ;
* arbitrages techniques ;
* communication technique ;
* gestion de complexité.

Un chef de projet, un RH, ou même un ancien développeur déconnecté du terrain peut difficilement mesurer cela avec précision.

Ce n’est pas une question de hiérarchie ; c’est une question de **légitimité d’évaluation**.

Dans les équipes matures, l’évaluation technique est souvent portée par :

* un lead dev ;
* un staff engineer ;
* un principal engineer ;
* parfois plusieurs pairs.

Le manager évalue autre chose :

* alignement ;
* communication ;
* organisation ;
* collaboration.

Mélanger les deux crée du bruit.

### 3. Le problème des tests techniques “algorithmico-logiques”

Beaucoup de tests évaluent :

* vitesse ;
* mémoire de patterns ;
* stress ;
* capacité à performer dans un format artificiel.

Mais développent-ils vraiment ceci ?

* conception de systèmes ;
* maintenabilité ;
* découplage ;
* observabilité ;
* résilience ;
* gestion de legacy ;
* collaboration ;
* debugging réel.

Souvent non.

Un développeur senior passe rarement ses journées à inverser un arbre binaire ou résoudre un problème de graphes.

Il passe son temps à :

* comprendre de l’existant ;
* limiter la complexité ;
* négocier des compromis.

---

### 4. Le mimétisme des modèles GAFAM

Beaucoup d’entreprises copient la forme sans comprendre la fonction.

Les grandes entreprises comme Google, Meta, Amazon ou Microsoft ont construit leurs process avec :

* énormément de données historiques ;
* des milliers de recrutements ;
* des corrélations statistiques ;
* des équipes spécialisées dans l’évaluation.

Leur objectif n’est pas de vérifier que vous connaissez “tel framework”.

Ils testent souvent :

* abstraction ;
* apprentissage ;
* adaptation ;
* raisonnement.

Beaucoup d’entreprises françaises reproduisent les exercices sans reproduire l’infrastructure intellectuelle qui les rend pertinents.

Résultat : du théâtre procédural.

---

### 5. Le biais élitiste

“Je vais vérifier si tu mérites d’entrer.”

au lieu de :

“Essayons de comprendre comment tu travailles.”

Le premier est un rapport de pouvoir.
Le second est un rapport d’évaluation.

La différence est énorme.

Une bonne interview cherche à réduire l’incertitude.
Une mauvaise cherche à établir une domination symbolique.

---

### Ce qui serait plus pertinent

Pour un développeur expérimenté :

**A. Revue de code réelle**
→ analyse d’un de vos projets

**B. Discussion d’architecture**
→ “comment auriez-vous conçu X ?”

**C. Debugging réel**
→ comprendre un bug existant

**D. Refactoring**
→ améliorer un code imparfait

**E. Pair programming**
→ observer le raisonnement en contexte

**F. Discussion postmortem**
→ parler d’un échec technique réel

Ces formats mesurent beaucoup mieux la réalité du métier.

---

**qu’est-ce qu’on cherche réellement à prédire chez un développeur ?**

Et tant que cette question n’est pas clairement posée, beaucoup de recrutements resteront médiocres — non pas faute de candidats, mais faute de méthode.
